### Accessibility audit

A formal pass against WCAG 2.1 AA:
- **Keyboard navigation:** Tab moves focus into and out of the editor (`tabIndents=false` recommended in docs); Escape moves focus out by default; no focus traps.
- **ARIA:** wrapper has accessible name (new prop `aria-label?: string` and/or `aria-labelledby?: string`); decorative widgets (bullets, checkboxes container, dividers) have `aria-hidden="true"`; interactive widgets (checkboxes themselves) have proper `aria-label`.
- **Color contrast:** every default light/dark variable pair meets WCAG AA contrast.
- **Reduced motion:** if any animations exist, respect `prefers-reduced-motion`.
- **Screen reader announcement:** test with VoiceOver and NVDA — content is read; widgets do not double-announce.
- **Forms:** the editor can be associated with a `<label>` via `aria-labelledby` or a wrapping `<label>`.

The audit is documented as `docs/accessibility.md` (new file): list of tested scenarios, results, known limitations.

Tooling: `axe-core` integration test on the demo app (one-off Playwright test or jsdom-based axe run); `@storybook/addon-a11y` enabled in stories.

### `tabIndents` default flip

In v0.4 we shipped `tabIndents` defaulting to `true` (legacy behavior). In v1.0 the default flips to `false` for accessibility. This is a breaking change but documented prominently in the migration guide.

### npmjs publish (optional)

The CI build is unchanged. The publish step uses `--registry=https://registry.npmjs.org` and an `NPM_TOKEN` instead of `CI_JOB_TOKEN`. The publish-to-public job runs only on tag pushes and only when a manual approval is given (`when: manual` in GitLab CI).

The `package.json` `publishConfig.registry` stays pointing at GitLab; the public publish overrides via CLI flag.

## Design notes

### Why a separate v1.0 milestone

We could fold "documentation polish" and "accessibility audit" into v0.6. We don't, because:
- Accessibility deserves its own focused pass — half-doing it during a feature release tends to miss things.
- Public release is a *commitment* (API stability, eventual npmjs publish). Treating it as a milestone with a real spec keeps the bar high.
- Documentation rewrite touches almost every file in `docs/`. Doing it in one focused PR (rather than scattered through v0.6) keeps the diff reviewable.

### Why optional public mirror / npmjs

These are decisions the maintainer makes when v1.0 ships, not now. Pre-committing now is premature. The spec calls them out explicitly so the maintainer remembers to decide.

### Why `tabIndents=false` becomes default

The rationale is purely accessibility: keyboard-only users must be able to escape the editor with Tab. Defaulting to `true` (current behavior) means every consumer must remember to override; defaulting to `false` means the conscious choice is to opt INTO indent-on-tab.

This breaks consumers (their editor stops indenting). The migration is one prop. We document it loudly.
