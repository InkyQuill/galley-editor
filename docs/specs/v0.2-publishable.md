## Design notes

### Why peerDependencies for CodeMirror

CodeMirror 6 packages (`@codemirror/state`, `@codemirror/view`, etc.) maintain identity-equality singletons internally. If a consumer installs a different version, or pulls in another library that itself depends on a different `@codemirror/view`, two copies of `EditorView` exist in memory and CodeMirror's plugin/decoration system silently breaks. By declaring them `peerDependencies` we force the consumer's package manager to deduplicate them.

Same reasoning applies to React (hooks contract requires single React identity).

### Why a separate `tsconfig.lib.json`

`tsconfig.app.json` has `noEmit: true` (correct for the demo app). The library build needs `.d.ts` emission. Rather than mutating the app config, a separate file lets us:
- Keep the app config clean.
- Exclude tests, stories, demo files from the typed output.
- Run `tsc -p tsconfig.lib.json` independently if we ever need to sanity-check declarations outside Vite.

The `vite-plugin-dts` plugin reads this config when emitting types.

### Why `sideEffects: ["**/*.css"]`

Bundlers (webpack/Rollup/esbuild) tree-shake more aggressively when `sideEffects: false`. But our CSS imports DO have side effects (registering styles in the document). Declaring `["**/*.css"]` tells bundlers: "every JS module is pure (tree-shakeable), but CSS imports are not — keep them." This preserves consumer tree-shaking of plugins/commands while still letting `import "@inky/neutrino-editor/style.css"` work.

### CI safeguard: `forbid-3rdparty-import`

A simple grep step in CI:

```sh
if grep -rE '(3rdparty|joplin|@joplin)' dist/ ; then
  echo "ERROR: Joplin reference material leaked into the build." >&2
  exit 1
fi
```

Defensive — the `tsconfig.lib.json` exclusion and `package.json` `files` allowlist already prevent this, but a third independent check costs nothing and protects against future configuration drift.
