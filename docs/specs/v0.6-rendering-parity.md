# v0.6.0 — Rendering Parity & Plugin API Polish

## Goal

The plugin/rendering layer matches the robustness of Joplin's editor. Reference-style links render. Bi-directional text works. Click-to-follow on links works. The plugin spec API is cleaned up so tuple-arity range ambiguity cannot create another multi-line block regression. A `useNeutrino()` hook lands as a hooks-first alternative to the imperative ref API.

After this milestone the rendering surface is feature-complete for v1.0.

## Scope

### Plugin spec API revision

The current `NeutrinoPluginSpec.getDecorationRange()` returning `[number] | [number, number] | null` carries decoration-type semantics in tuple arity. v0.3 `makeBlockPlugin` correctly expands line decorations across every line in a returned range, but the API still makes tuple length stand in for decoration intent. The new API explicitly separates the three cases:

```ts
interface NeutrinoPluginSpec {
  /** Produce a decoration to apply, or null to skip. */
  createDecoration(
    node: SyntaxNodeRef,
    state: EditorState,
    parentDepths: ReadonlyMap<string, number>,
  ): WidgetType | Decoration | null;

  /** Range for line decorations (Decoration.line applied to every line in [from..to]). */
  getLineRange?(node: SyntaxNodeRef, state: EditorState):
    | { from: number; to: number }
    | null;

  /** Range for mark / replace decorations. */
  getMarkRange?(node: SyntaxNodeRef, state: EditorState):
    | { from: number; to: number }
    | null;

  /** Position for point decorations (widgets). */
  getPointPosition?(node: SyntaxNodeRef, state: EditorState): number | null;

  /** When to show raw markdown instead of the decoration. Default 'line'. */
  getRevealStrategy?(node: SyntaxNodeRef, state: EditorState): RevealStrategy;

  /** Whether cursor proximity hides the decoration. Default true. */
  hideWhenNearCursor?: boolean;

  /** Performance: skip rebuilds when this returns false. */
  selectionAffectsDecorations?(prev: EditorSelection, next: EditorSelection): boolean;

  /** Force full re-render on specific transactions. */
  shouldForceRerender?(transaction: Transaction): boolean;
}
```

**Validation:** at most one of `getLineRange` / `getMarkRange` / `getPointPosition` may be defined per spec. The factory throws at construction time if multiple are present.

**Migration of built-in plugins:** every plugin in `src/plugins/` is rewritten to use the new shape. Mostly mechanical (rename `getDecorationRange` to whichever matches, return objects instead of tuples).

**No back-compat alias.** The old `getDecorationRange` is removed (we have permission to break pre-1.0).

### Reference-style links

**Behavior specification:**

A reference-style link in markdown looks like:

```markdown
This is a [link to something][ref].

[ref]: https://example.com "Optional title"
```

The current `linksPlugin` only handles inline links (`[text](url)`). The new plugin:
1. Detects `[text][ref]` syntax (inline reference).
2. Detects `[ref]` shorthand syntax (when the link text and ref label are the same).
3. Detects `[ref]: url "title"` definitions and registers them in a state field (the registry).
4. Renders the inline-reference span with `ne-link` class, like inline links.
5. Hides the `[ref]` part (and the `:url` definition line) using existing reveal strategies.
6. On link click (Cmd/Ctrl-click), opens the resolved URL.

**Implementation:**
- New `src/plugins/links-reference.ts` separate from `links.ts` for clarity, or fold into `links.ts` if cleaner.
- New state field `linkRegistryField` holds `Map<label, { url, title }>` updated on every doc change.
- The Link mark plugin consults the registry to find the URL for the click handler.

### Click-to-follow on links

**Behavior:** Cmd-click (macOS) / Ctrl-click (other) on a `Link` decoration opens the URL.

**Implementation:** add a `mousedown` (or `click`) handler in `linksPlugin` that:
1. Checks if `event.metaKey` (mac) or `event.ctrlKey` (other) is set.
2. Locates the `Link` node at the click position via `view.posAtCoords` + syntax tree.
3. Reads the URL — for inline links, from the URL node; for reference links, from the registry.
4. `window.open(url, '_blank', 'noopener,noreferrer')`.

**Configurable behavior:** new prop `onLinkClick?: (url: string, event: MouseEvent) => boolean | void`. If defined, called instead of `window.open`. Returning `true` means "consumer handled it, don't fall back". Returning `void` / `false` means "fall back to default open".

### Bi-directional text support

**Behavior specification:**

Text containing right-to-left scripts (Arabic, Hebrew, etc.) renders correctly with proper bidi isolation. Specifically:
- Lines containing primarily RTL text render right-aligned with cursor movement following the visual direction.
- Mixed LTR/RTL lines use Unicode bidi rules.
- The Decoration system inserts `dir="auto"` on relevant elements so the browser's bidi algorithm runs.

**Implementation:** small `biDirectionalTextExtension` that:
1. Adds `dir="auto"` to every `.cm-line` element via a `Decoration.line({ attributes: { dir: 'auto' } })`.
2. No JS bidi algorithm — we rely on the browser's `dir="auto"`.

This is a feature toggle: new prop `bidi?: boolean` (default `false` — bidi rendering can subtly affect layout for LTR-only documents). Documented in `docs/api-reference.md`.

### Image rendering hook

**Behavior specification:**

By default, `![alt](url)` renders as the rendered text "alt" with the URL hidden (similar to inline links). With `imageRenderer` prop, consumers can replace this with a custom widget — e.g. an actual `<img>` thumbnail.

**Prop:**
```ts
imageRenderer?: (image: { alt: string; url: string; title?: string }) => HTMLElement | null;
```

If provided, the `imagesPlugin` (new file `src/plugins/images.ts`) calls the renderer for each image node. If the renderer returns null or no prop is provided, fall back to text-with-hidden-URL (same shape as the current Link rendering).

**Default behavior:** no rendering of actual `<img>` (consumers can opt in). Avoids accidentally fetching arbitrary URLs.

### Performance: selective rebuilds

`makeInlinePlugin` currently rebuilds on every `selectionSet`, even when the cursor moved within the same word with no decoration impact. The new API includes:

```ts
selectionAffectsDecorations?(prev: EditorSelection, next: EditorSelection): boolean;
```

Default: `() => true` (current behavior). Plugins like `emphasisPlugin` that only care about whether the cursor is inside a `StrongEmphasis` node can implement smarter logic and skip rebuilds when the cursor moves but stays in the same node.

### `useNeutrino()` React hook

Alternative API for hooks-first consumers:

```tsx
function MyEditor() {
  const { ref, content, setContent, focus, execCommand } = useNeutrino({
    initialValue: '# Hello',
    onChange: (v) => console.log(v),
  });

  return <NeutrinoEditor ref={ref} />;
}
```

The hook:
- Owns a `ref` (typed `NeutrinoHandle | null`).
- Returns the imperative methods as memoized stable function references.
- Returns the latest `content` as state (reactive).
- Internally subscribes to `onChange` on mount.

**File:** `src/hooks/useNeutrino.ts`. Exported from the package barrel.

This is purely additive — the imperative `ref` API stays as the primary interface.

## Out of scope

- Math/LaTeX rendering (consumers add via `extensions` prop or KaTeX).
- Mermaid diagram rendering (same).
- Syntax highlighting inside fenced code blocks (consumers add via CodeMirror language packs).
- Footnotes (markdown extension; not GFM core).
- Wikilinks (`[[Page Name]]` syntax — Obsidian-specific).
- Custom token highlighting plugin (the `classHighlighter` from CM6 is enough).

## Design notes

### Why split `getDecorationRange` into three methods

The current API's tuple-arity-as-type-discriminator is the kind of cleverness that bites you. It's not obvious from the signature that:
- `[from]` for a line decoration means "decorate the line containing position `from`".
- `[from]` for a mark decoration means "create a zero-width mark at `from`" (which you almost never want).
- `[from, to]` for a line decoration means "decorate every line in `[from..to]`", while the same tuple shape means a continuous range for mark and replace decorations.

Three methods, three names, three return shapes. No discriminated unions; no tuple-arity.

### Why click-to-follow needs a prop

Auto-opening URLs is an opinionated default. Single-page-app consumers might prefer `react-router`'s navigate; embedded-editor consumers might want to log analytics first. The `onLinkClick` prop gives them control without the lib being prescriptive.

### Why `useNeutrino` is additive

The imperative ref API is the primary surface for two reasons:
1. It composes naturally with non-React code (a vanilla JS app could use the same `EditorController`).
2. It's explicit about lifecycle — refs are nullable, and methods are direct.

The hook is sugar for the common React case. It uses the ref under the hood; consumers needing both can use the hook AND grab the ref it returns.
