# v0.3.0 — Correctness & Theming

## Goal

Every CRITICAL and HIGH finding from the pre-spec code review is fixed and pinned with a regression test. The `theme` prop actually changes appearance. Theming is fully driven by CSS custom properties; consumers can theme via plain CSS overrides, Tailwind v4 `@theme`, or `classNames` prop overrides. After this milestone the library is functionally production-grade; subsequent milestones add features rather than fix bugs.

## Status

**in-progress**

## Scope

### Bug fixes — runtime correctness

Each item is a known bug from the review. Each ships with a regression test.

#### B1. Multi-line block decorations

**Problem:** `blockquotePlugin` and `tablesPlugin` apply their line decoration only to line 1 of multi-line blocks. `getDecorationRange` returns `[line.from]` (a single position), and `makeBlockPlugin` interprets a 1-tuple as a single-line decoration.

**Fix:** in `rendering.ts`, when a block plugin's `createDecoration` returns a `Decoration.line` decoration, iterate every line in `[node.from, node.to]` and emit the line decoration for each. The `getDecorationRange` 1-tuple/2-tuple distinction is preserved for non-line decorations (widgets), but for line decorations we always span the full block.

**Test:** `plugins/blockquote.test.ts` — three-line blockquote, assert all three lines have the `ne-blockquote` class. Same for `plugins/tables.test.ts`.

#### B2. `editorClassName` reactivity

**Problem:** `editorClassName` is added once on mount via direct DOM mutation; prop changes are silently ignored.

**Fix:** introduce an `editorClassNameCompartment` in `EditorController`. Its extension is a `ViewPlugin` that diffs the desired class list against `view.dom.classList`, adds new entries, removes departed entries. `updateSettings` reconfigures this compartment when the prop changes.

**Test:** `components/NeutrinoEditor.test.tsx` — render with `editorClassName="a"`, assert class on `.cm-editor`; rerender with `editorClassName="b"`, assert `a` is gone and `b` is present; rerender with empty string, assert both are gone.

#### B3. Imperative handle null guard

**Problem:** every method uses `controllerRef.current!` non-null assertion. Throws if a parent's `useLayoutEffect` calls the ref before the editor's mount `useEffect` runs.

**Fix:** every method becomes nullable-safe via `?.`. Methods returning `void` no-op silently. Methods returning a value return a sensible default (`getContent()` → `""`, `getSelection()` → `{from:0,to:0,anchor:0,head:0}`). The `view` getter changes type to `EditorView | null`. This is a breaking type change but acceptable pre-1.0.

**Test:** `components/NeutrinoEditor.test.tsx` — render via React Testing Library with a parent that calls `ref.current.focus()` from its own `useLayoutEffect`; assert no throw.

#### B4. Escape passthrough

**Problem:** Escape keymap returns `true` even when no `onEscape` handler is registered, swallowing the event for consumer extensions and outer modal handlers.

**Fix:** the Escape handler returns `true` only when `onEscape` is defined and explicitly returns truthy. If `onEscape` is `undefined` or returns `void`/`false`, the handler returns `false` so other extensions / parent components see the event.

**Type change:** `onEscape` signature becomes `() => boolean | void` (was `() => void`). Returning `true` consumes the event; void or false lets it bubble.

**Test:** `controller.test.ts` — dispatch Escape with no handler, assert event `defaultPrevented` is false; with handler returning true, assert prevented.

#### B5. `Mod-Shift-Enter` should not fire `onSubmit`

**Problem:** `handleEnter(_, mod=true, shift=true)` fires `onSubmit` because the check ignores `shift`.

**Fix:** `onSubmit` fires only when `mod && !shift`.

**Test:** `controller.test.ts` — dispatch Mod+Shift+Enter, assert onSubmit not called; dispatch Mod+Enter, assert called once.

#### B6. H1 heading toggle replaces existing levels

**Problem:** `createToggleHeading(1)` skips the "remove other levels" pass (`if (level > 1)` guard), producing `# ## hello` from `## hello`.

**Fix:** remove the `level > 1` guard. Level 1 also removes existing heading levels first.

**Test:** `commands.test.ts` — update the now-failing test (`toggleHeading1 replaces ## with #`) to expect `# hello`. Add the symmetric test for every level transition.

#### B7. `setContent` preserves selection

**Problem:** `setContent` clamps the old anchor against new length and installs a single-cursor selection, losing direction and any secondary cursors.

**Fix:** when the new content is set, attempt to preserve the main selection's direction (`anchor`, `head`) by clamping each independently, and preserve any additional selection ranges by clamping each to `[0, newLength]`. If a range collapses to zero width, keep it as a cursor.

**Test:** `controller.test.ts` — set up multi-cursor (3 ranges), call `setContent` with shorter content, assert all three cursors clamped and direction preserved where possible.

#### B8. `addExtension` no-leak

**Problem:** uses `StateEffect.appendConfig` which permanently adds a slot; `compartment.reconfigure([])` empties the slot but the slot stays. 100 add/remove cycles = 100 dead slots.

**Fix:** introduce a single `runtimeExtensionsCompartment` (created in the constructor), holding the result of computing `Array.from(registry.values())`. `addExtension(ext)`:
1. Generate a unique symbol key.
2. Insert into the registry map.
3. Reconfigure the compartment.
4. Return `{ remove: () => { registry.delete(key); reconfigure(); } }`.

No `appendConfig`, no leaks.

**Test:** `controller.test.ts` — add 100 extensions, remove all, dispatch a transaction, verify state config doesn't grow unboundedly (no easy public API for this; we test by counting: store initial config length, add 50, remove 50, assert final length equals initial).

#### B9. `parentDepths` reset between visible ranges

**Problem:** in `makeInlinePlugin`, `parentDepths` is allocated once outside the `for (const { from, to } of view.visibleRanges)` loop. Counts leak between ranges; bullet/checkbox depth math is wrong on viewports with multiple visible ranges.

**Fix:** allocate a fresh `parentDepths` map inside the `for` loop body, before each `iterate()` call.

**Test:** hard to provoke in jsdom (visible ranges are always one). Instead, write a unit test that manually invokes the iteration helper with two consecutive ranges and asserts the second range starts with empty depths. (Refactor required: extract the iteration body into a testable helper.)

#### B10. `BulletMarkerWidget.updateDOM` previous-class tracking

**Problem:** removes hard-coded literals `'ne-depth-0', 'ne-depth-1', 'ne-depth-2'` regardless of actual previous state. Breaks if `DEPTH_STYLE_COUNT` changes or if the marker class is overridden.

**Fix:** the widget instance stores `this.previousDepthClass` (set in `toDOM`/`updateDOM`). `updateDOM` removes that exact class and adds the new one.

**Test:** `plugins/lists.test.ts` — render bullet marker, simulate prop change to a new depth, assert old depth class removed and new added.

#### B11. Decorative widget aria-hidden

**Problem:** `BulletMarkerWidget` declares `role="img"` (announced as "image" by screen readers, redundant with the list item itself). `DividerWidget`'s `<hr>` has no `aria-hidden`, so SR double-announces "separator" alongside the (hidden) source text.

**Fix:** both widgets get `aria-hidden="true"` and drop the `role` attribute. The list semantics come from the underlying list item; the divider semantics come from the markdown source.

**Test:** `plugins/lists.test.ts` and `plugins/dividers.test.ts` — render, assert `aria-hidden="true"` on the widget DOM.

#### B12. CheckboxWidget surgical edit

**Problem:** clicking the checkbox dispatches a transaction that replaces the entire line. Cursors and selections elsewhere on that line are invalidated.

**Fix:** locate the precise byte range of the marker (the `[ ]` or `[x]` substring) within the line. Dispatch a change replacing only those 3 characters, preserving everything else.

**Test:** `plugins/checkboxes.test.ts` — set up cursor at end of line, click checkbox programmatically (or simulate via direct widget invocation), assert cursor still at end of line (offset adjusted for the replacement length, which is zero).

#### B13. Multi-cursor handling in `handleEnter`

**Problem:** `handleEnter` operates on `selection.main` only. Secondary cursors don't get newlines.

**Fix:** rewrite `handleEnter` to use `state.changeByRange((range) => ...)`, returning a change spec for each range. Apply uniformly.

**Test:** `controller.test.ts` — set up 2 cursors, press Enter, assert both lines split.

#### B14. Event coalescing for `onScroll` / `onSelectionChange`

**Problem:** these fire on every micro-event, flooding consumer state.

**Fix:** introduce a `rafCoalesce(fn)` helper. Wrap `onScroll` and `onSelectionChange` callbacks in the coalescer (one call per animation frame). The selection change still reports the *latest* selection at frame end.

**Test:** `controller.test.ts` — dispatch 10 selection changes synchronously, advance one rAF tick, assert callback called exactly once with the final selection.

#### B15. `autosizeExtension` first-measure stability

**Problem:** uses `view.defaultLineHeight || 24` which can be 0 before first measure; can ping-pong on the boundary.

**Fix:**
- Use `view.requestMeasureSync` (or `requestMeasure` async) for the first measurement.
- Track the previous target height; skip the style write if the target hasn't changed by more than 1 pixel.

**Test:** `autosize.test.ts` — verify height stabilizes within 2 update cycles for a doc shorter than `minRows`; for a doc longer than `maxRows` the scroller becomes scrollable.

#### B16. Drop dead `updateCallbacks` method

The callback proxy pattern already solves the staleness problem. Remove the unused method from `EditorController`.

#### B17. Storybook Tailwind enabled in preview

**Problem:** `.storybook/main.ts` `viteFinal` has `css: { postcss: { plugins: [] } }` which disables PostCSS/Tailwind in stories.

**Fix:** drop the `css.postcss` override so Storybook picks up the project's `postcss.config.js` (which has `@tailwindcss/postcss`).

**Implementation note:** verified as already fixed in v0.2; no additional v0.3 runtime change was required.

#### T2. CodeMirror theme reads from the same variables

`src/theme.ts` is rewritten. `buildCmTheme(scheme)` returns a CM6 theme object whose values are `'var(--ne-color-caret)'` etc. — the CM6 theme system supports CSS values, so `caretColor: 'var(--ne-color-caret)'` works.

This means dark caret/selection/focus-ring follow `data-theme="dark"` automatically.

#### T3. `data-theme` attribute on the wrapper

`NeutrinoEditor.tsx` renders:

```tsx
<div className={className} data-theme={resolvedTheme}>
  <div ref={containerRef} />
</div>
```

`resolvedTheme` is `'light' | 'dark'`, computed by the controller (which subscribes to `matchMedia` for `'auto'`).

#### T4. Live OS-theme subscription for `theme="auto"`

`EditorController` constructor accepts an optional theme-resolved callback. When `theme === 'auto'`, the controller calls `window.matchMedia('(prefers-color-scheme: dark)')` and subscribes to its `change` event. On change, the callback fires; the React component sets state, the controller's settings update, the wrapper's `data-theme` attribute flips.

The subscription is cleaned up in the controller's `destroy` method.

**SSR safety:** the matchMedia call is guarded by `typeof window !== 'undefined'`. Initial render returns `'light'`. After hydration, the effect runs and corrects the attribute. Brief flash possible — acceptable for a non-SSR-first library.

#### T5. Tailwind v4 integration recipe

Document, with tested code, in `docs/styling.md`:

```css
/* consumer's app.css */
@import "tailwindcss";
@import "@inky/neutrino-editor/style.css";

@theme {
  /* Map editor variables to Tailwind tokens */
  --color-ne-text: var(--color-gray-900);
  --color-ne-link: var(--color-blue-600);
  --color-ne-code-bg: var(--color-gray-100);
  /* ... */
}

[data-theme="dark"] {
  /* dark overrides also map to Tailwind */
  --color-ne-text: var(--color-gray-100);
  --color-ne-link: var(--color-blue-400);
  /* ... */
}
```

Tailwind v4 (`@theme` block) generates the underlying `--color-*` tokens, which our `--color-ne-*` aliases reference. The doc walks through the full mapping.

#### T6. `classNames` prop kept

The `classNames` prop continues to work for consumers who want to replace `ne-*` classes entirely (e.g., to use their own design system). The CSS variables are the recommended path; class replacement is the escape hatch.

## Design notes

### Why CSS variables instead of CSS-in-JS

Three reasons:
1. **Consumer override is one line of CSS.** No need to know the component's prop API, no Tailwind plugin needed for theming.
2. **Tree-shaking-friendly.** No runtime CSS generation, no theme provider wrapping, no extra render passes.
3. **Tailwind v4 native.** v4's `@theme` produces `--color-*` tokens, which our variables compose with via `var()`. The bridge is trivial.

### Why `data-theme` attribute instead of CSS class

Both work; we pick `data-theme` because:
- It's an explicit two-state attribute (`light` / `dark`), not a sometimes-present class.
- It composes naturally with `[data-theme="dark"]` selectors without leaking into Tailwind's `dark:` variant (which uses a class). Consumers using Tailwind's dark mode keep using `dark:` for their own components; we don't conflict.
- Easy to debug — `inspect → check the attribute`.

### Why a separate compartment for `editorClassName`

We could mutate `view.dom.classList` directly in the React effect. We don't, because:
- Mutations outside CodeMirror's transaction system can desync from CM6's expectations (rare but messy).
- A compartment makes the operation symmetric: `updateSettings` re-applies a single source of truth.
- The compartment can later carry related view-DOM concerns (e.g., `aria-label`) without growing surface area.
