# v0.4.0 — Smart Input

## Goal

Typing in the editor *feels* alive. Enter inside a list creates the next item; Tab nests; Shift-Tab outdents; Backspace at the start of a list marker exits the list. The `commands.ts` module is rewritten to use `state.changeByRange` for proper multi-cursor support and to fix corner cases the original implementation got wrong. Common keyboard shortcuts (Mod-K for link, Mod-D for duplicate) become first-class.

After this milestone the editor does what every modern markdown editor does on Enter/Tab. The README's pre-existing (and unfulfilled) "smart list continuation" claim is finally true.

## Scope

### Smart Enter — list continuation

**Behavior specification** (written from observed Joplin behavior, not from Joplin source):

When the cursor is at end of a non-empty list-item line and Enter is pressed:
1. Detect the kind of list (`-`, `*`, `+`, `1.`, `- [ ]`, `- [x]`).
2. Compute the next marker:
   - Bulleted (`-`, `*`, `+`): same marker.
   - Ordered (`1.`): next integer (but the markdown renderer ignores the actual numbers, so we always emit `next-int.`).
   - Task (`- [ ]`): always `- [ ]` (next task is unchecked, even if previous was checked).
3. Insert `\n` followed by the same indent as the current line, followed by the next marker, followed by a single space.
4. Move the cursor after the marker.

When the cursor is at end of an *empty* list-item line (line text is exactly the marker, possibly with trailing space):
1. Replace the marker with empty (delete the line content).
2. Move the cursor to the start of the (now-empty) line.
3. The next Enter press creates a normal newline.

When the cursor is in the middle of a list-item line:
- Same as end-of-line case, but the text after the cursor moves to the new line, prefixed with the new marker.

When the cursor is *not* in a list:
- Default Enter behavior (insert `\n`, no marker handling). This is the existing `handleEnter` logic, retained.

**Edge cases:**
- Indented sub-lists: indentation is preserved.
- Mixed lines: if the cursor is on a non-list line directly after a list, default behavior.
- Multi-cursor: each range processed independently via `changeByRange`.
- Selection (non-empty): if a selection spans inside a list item, replace the selection with `\n + marker` (collapses to the new-line case).

**Implementation:** new file `src/commands/smartEnter.ts`. The existing `handleEnter` in `controller.ts` calls this first; if no list-continuation applies, falls through to the existing logic (which will become `defaultEnter`).

### Smart Tab / Shift-Tab in lists

**Behavior specification:**

When the cursor (or selection start) is on a list-item line:
- **Tab**: increase indentation by one indent unit. The marker moves with the indent.
- **Shift-Tab**: decrease indentation by one indent unit. If already at column 0, no-op (or signal "fall through to default").

When not on a list line:
- Tab: insert indent unit (existing behavior, controlled by `tabIndents` prop — see §Keymap below).
- Shift-Tab: no-op or fall through.

**Edge cases:**
- Multi-line selection: every selected line indents/outdents together.
- Numbered lists: nesting an ordered list resets the number to `1.` (matches Obsidian behavior).
- Task lists: indentation preserves the `[ ]` / `[x]` state.

**Implementation:** new file `src/commands/smartTab.ts`. Bound to `Tab` and `Shift-Tab` keys with higher precedence than `indentWithTab`.

### Smart Backspace at list marker

**Behavior specification:**

When the cursor is positioned exactly after a list marker on an otherwise-empty content (e.g., `- |` where `|` is the cursor) and Backspace is pressed:
1. Delete the entire marker including the trailing space.
2. The line becomes a plain paragraph at column 0.

When the cursor is anywhere else: default Backspace.

**Edge cases:**
- Indented marker: Backspace removes the marker but preserves indentation, leaving an indented empty line. (Then a second Backspace at column 0 of an indented line outdents — handled by smartTab logic if we want, or by default.)
- Inside a task `- [ ]`: Backspace removes `- [ ] ` entirely.

**Implementation:** new file `src/commands/smartBackspace.ts`.

### Rewrite of `commands.ts`

The current `commands.ts` has issues beyond the v0.3 H1 fix:
- Inline formatting commands don't use `changeByRange` properly for multi-cursor.
- `indentCmd` regex `/$ ^/` is unreadable (works only by always-not-matching).
- Duplicated logic between heading-toggle commands.

**Rewrite strategy:**
- Split into a directory `src/commands/`:
  - `inline.ts` — toggleBold, toggleItalic, toggleCode, toggleStrikethrough.
  - `block.ts` — toggleHeading (single function, takes level 1-6), toggleBulletList, toggleOrderedList, toggleCheckList.
  - `insert.ts` — insertLink, insertImage, insertCodeBlock, insertTable, insertHr.
  - `editing.ts` — indent, outdent, undo, redo, selectAll.
  - `smartEnter.ts`, `smartTab.ts`, `smartBackspace.ts` (above).
  - `index.ts` — exports `BUILTIN_COMMANDS` registry.
- All commands use `state.changeByRange((range) => ({ changes, range: newRange }))` for proper multi-cursor.
- Heading: single command `toggleHeading(level: 1 | 2 | 3 | 4 | 5 | 6)`. The per-level commands (`toggleHeading1` through `toggleHeading6`) are removed (no consumers; we have permission to break). The `BuiltinCommand` type is updated.
- Ordered-list toggle: when toggling on, renumber the inserted items as 1., 2., 3., ... (current code emits `1. ` for every item — markdown renderers don't care, but the visual source looks wrong).
- Insert commands place the cursor sensibly:
  - `insertCodeBlock`: cursor on the empty line between the fences.
  - `insertTable`: cursor on the first content cell (between the pipes).
  - `insertLink` with no label: cursor inside the `[]`.
  - `insertHr`: cursor on the line after the `---`.

### Default keymap

The default keymap is reorganized and made overridable:

| Key | Action |
|---|---|
| `Enter` | Smart Enter (with default fallback) |
| `Shift-Enter` | Default newline (bypass smart) |
| `Tab` | Smart Tab in lists; otherwise indent if `tabIndents=true` |
| `Shift-Tab` | Smart Shift-Tab in lists; otherwise outdent |
| `Backspace` | Smart Backspace; otherwise default |
| `Mod-Enter` | `onSubmit` callback |
| `Mod-Shift-Enter` | (reserved; falls through) |
| `Escape` | `onEscape` callback (returns `false` to allow bubbling unless callback returns `true`) |
| `Mod-B` | `toggleBold` |
| `Mod-I` | `toggleItalic` |
| `Mod-K` | `insertLink` (with cursor inside `[]`) |
| `Mod-D` | `duplicateLine` (added in v0.5; reserved here) |
| `Mod-Z` / `Mod-Shift-Z` | undo / redo (existing) |
| `Mod-A` | selectAll (existing) |

**`tabIndents` prop:** new boolean prop on `NeutrinoEditor`, default `true`. When `false`, Tab is allowed to move focus out of the editor (accessibility-first).

**`keymap` prop:** new optional prop, type `KeyBinding[] | (defaults: KeyBinding[]) => KeyBinding[]`. The function form lets consumers extend or override.

## Design notes

### Why we re-derive the "smart" commands instead of porting Joplin's

Joplin's `insertNewlineContinueMarkup.ts` is well-designed and battle-tested. We *could* port it. We don't, for two reasons:
1. **Clean-room policy.** This is the entire point of the license boundary. We write from a behavior spec; we re-discover the algorithm.
2. **The algorithm is small.** Smart-Enter for markdown lists is ~50 lines of careful tree-iteration logic. Re-deriving costs hours, not days, and produces code we fully understand.

The behavior spec in this document is the *result* of reading Joplin's code with the source closed afterwards.

### Why a single `toggleHeading(level)` instead of six commands

The six commands are an anti-pattern: they share 95% logic, the level is data not a separate command, and a toolbar UI naturally takes a level argument from a dropdown. The new shape:

```ts
ref.current.execCommand('toggleHeading', 2)  // H2
```

The `BuiltinCommand` union becomes 17 entries instead of 23.

### Multi-cursor consistency

Every command in the new `commands/` directory uses `state.changeByRange`. This makes them all multi-cursor-correct by construction. The few existing tests that assume single-cursor still pass because `changeByRange` with a single range degenerates to the same result.

### Why `Shift-Enter` is escape hatch

Some consumers want Enter inside a list to *not* continue the marker (e.g., a chat-like UI where Enter submits and a list inside a draft should not auto-continue). Shift-Enter as the "default newline" gives them an escape hatch without disabling the smart behavior wholesale.

### `tabIndents=false` accessibility default

For v0.4 the default is `true` to preserve current behavior. We discuss flipping the default to `false` in v1.0 as part of the accessibility audit.
