# v0.5.0 — Editor Commands

## Goal

Ship the productivity commands every modern editor has: duplicate line, sort lines, swap lines, jump to anchor, basic find. The keymap becomes properly customizable via a prop. Commands are well-documented in `docs/commands.md`.

After this milestone the editor matches the productivity affordances of Obsidian / VS Code's "lite" command palette, without bringing in a search-panel UI or any of the heavier machinery.

## Scope

### New commands

Every command in this milestone is authored from a written behavior spec. Implementation lives in `src/commands/editing/` (new subdirectory).

#### `duplicateLine`

**Behavior:** copy the current line (or every line containing a selection range), insert it immediately after. Cursor follows to the duplicated line. Multi-cursor: each range duplicates its own line(s).

**Default keybinding:** `Mod-D`. (CodeMirror's default `Mod-D` is "select next occurrence" — we override.)

**File:** `src/commands/editing/duplicateLine.ts`.

#### `sortSelectedLines`

**Behavior:** sort the lines that the selection touches, in place. Default sort is locale-aware ascending. No keybinding by default; available via `execCommand('sortSelectedLines')`. Optional argument `direction: 'asc' | 'desc'` (default `'asc'`).

Single-cursor (no selection): no-op (or sort the whole document if invoked with an explicit `wholeDocument: true` flag).

**File:** `src/commands/editing/sortSelectedLines.ts`.

#### `swapLineUp` / `swapLineDown`

**Behavior:** swap the current line with the line above (or below). Cursor moves with the line. Multi-cursor: each range swaps independently, but if two ranges' lines collide (e.g., adjacent cursors swapping in conflicting directions), the lower-numbered cursor wins and the other no-ops.

**Default keybindings:** `Alt-ArrowUp` / `Alt-ArrowDown` (matches VS Code).

**Files:** `src/commands/editing/swapLine.ts` (both directions in one file).

#### `insertLineAfter` / `insertLineBefore`

**Behavior:** insert a new line after (or before) the current line, position cursor at start of the new line. Useful for creating space without needing End/Home keystrokes.

**Default keybindings:** `Shift-Enter` (already used for "literal newline") *conflicts* — we use `Mod-Alt-ArrowDown` and `Mod-Alt-ArrowUp` instead. Or no default; available via `execCommand`.

**Files:** `src/commands/editing/insertLine.ts`.

#### `jumpToHash(hash: string)`

**Behavior:** scan the document for an anchor matching `#hash` (markdown heading IDs follow GitHub's slug convention: lowercase, `-` for spaces, strip punctuation). On match, scroll to that line and place the cursor there. On no match, return `false`.

Useful for in-document linking and for consumers who want to wire `<a href="#some-section">` to the editor.

**Signature:** `(hash: string) => boolean` — returns whether a match was found.

**File:** `src/commands/navigation/jumpToHash.ts`.

#### `findInDocument(needle: string, opts?: FindOpts): FindResult[]`

**Behavior:** plain-text scan (no UI, no panel). Returns an array of match positions. Optional flags: `caseSensitive`, `wholeWord`, `regex`. No replace functionality — this is a scaffolding API; consumers build their own UI on top.

**Signature:**
```ts
type FindOpts = { caseSensitive?: boolean; wholeWord?: boolean; regex?: boolean };
type FindResult = { from: number; to: number; line: number };
function findInDocument(view: EditorView, needle: string, opts?: FindOpts): FindResult[];
```

Available via `execCommand` and exposed as a named export from the package.

**File:** `src/commands/navigation/findInDocument.ts`.

### Keymap polish

The `keymap` prop was introduced in v0.4. v0.5 finalizes the surface:

- Export `DEFAULT_KEYMAP: KeyBinding[]` so consumers can reference / filter without using the function form.
- Add inline JSDoc on every default binding so IDE hover shows the action.
- Document the full default keymap table in `docs/commands.md`.

**Function form** (recommended) receives the full default keymap and returns a (possibly filtered/extended) keymap. This lets consumers add bindings without losing defaults:

```tsx
<NeutrinoEditor
  keymap={(defaults) => [
    ...defaults,
    { key: 'Mod-Shift-K', run: (view) => insertCallout(view) },
  ]}
/>
```

**Array form** replaces the entire keymap. Use only for total override (rare).

The compartment infrastructure (`keymapCompartment` in `EditorController`) was set up in v0.4 alongside the prop introduction. v0.5 changes here are limited to the documented exports above; no controller changes.

### Command registry export

`BUILTIN_COMMANDS` becomes the canonical lookup. `execCommand(name, ...args)` looks up the function and calls it. New behavior:
- Custom-registered commands (via `registerCommand`) take precedence over built-ins (already the case).
- Unknown command name: returns `undefined` and logs a console warning (already the case).
- Built-in command names exported as a TypeScript const for autocomplete:
  ```ts
  export const BUILTIN_COMMAND_NAMES = ['toggleBold', 'toggleItalic', /* ... */] as const;
  ```

### `docs/commands.md`

A new user-facing doc lists every built-in command:
- Name
- What it does (one paragraph)
- Default keybinding (if any)
- Arguments (if any)
- Return value (if non-void)
- Code example

Generated as a starter from inline JSDoc on each command function; finalized by hand.

## Out of scope

- Search panel UI (consumers can build on top of `findInDocument`).
- Vim mode (out of scope entirely; consumers use `@replit/codemirror-vim` via `extensions`).
- Replace functionality (consumers can implement on top of `findInDocument` + `view.dispatch`).
- Command palette UI (out of scope; an example Storybook story may demonstrate building one).

## Design notes

### Why no built-in search panel

CodeMirror has `@codemirror/search` which provides a complete panel UI. Consumers who need it pass it via the `extensions` prop. We don't ship it because:
- Adds ~15kb gzipped to the lib for a feature many consumers won't use.
- Search-panel UI is opinionated (positioning, styling); consumers often want their own.
- `findInDocument` covers the *programmatic* need; the UI is a presentation concern.

### Why `Mod-D` for duplicateLine instead of "select next"

CodeMirror's default `Mod-D` is "select next occurrence of selection" — useful for multi-cursor refactoring. But:
- For a markdown editor, "duplicate line" is the more frequently-needed action.
- Consumers who want "select next" can rebind via the `keymap` prop.

This is a deliberate divergence from CodeMirror's defaults, documented in `docs/commands.md`.

### `findInDocument` returns positions, not a Cursor object

Returning a plain array makes the result easy to render (e.g., highlight all matches with a `Decoration` consumer-side). A `Cursor` would be statefully iterable but harder to consume from React. The cost is allocating an array; for typical doc sizes this is negligible.
