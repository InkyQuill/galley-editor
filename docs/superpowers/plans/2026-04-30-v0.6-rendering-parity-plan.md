# v0.6 Rendering Parity Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [x]`) syntax for tracking.

**Goal:** Ship the v0.6.0 rendering-parity milestoge: explicit plugin range APIs, reference links, link activation hooks, opt-in bidi support, image renderer customization, selective decoration rebuilds, and `useGalley()`.

**Architecture:** Keep the existing CodeMirror 6/Lezer architecture and compartment-based controller. Make the generic rendering factories stricter and clearer, migrate built-ins to the new spec shape, then layer feature-specific state fields/extensions for reference links, bidi line attributes, image rendering context, and hook ergonomics.

**Tech Stack:** TypeScript, React 19, CodeMirror 6, Lezer Markdown/GFM, Vitest/jsdom, Storybook, CSS semantic classes.

---

## Resolved Spec Questions

- `imageRenderer` is synchronous: `(image) => HTMLElement | null`. Async rendering can be implemented by consumers with their own plugin or by returning an element that manages its own async lifecycle.
- `bidi` remains an opt-in prop with default `false`, matching the spec and avoiding subtle layout shifts for LTR-only documents.
- Link following uses a `click` handler. It is the standard activation event, and it avoids hijacking normal selection changes during `mousedown`.
- Image default behavior changes from the current v0.5 actual `<img>` widget to the v0.6 spec's safer default: show alt text/link-like text and hide URL syntax. Actual thumbnails require `imageRenderer`.

## Current-State Notes

- `imagesPlugin` currently renders actual `<img>` widgets by default. v0.6 intentionally changes this behavior and updates tests/Storybook/docs accordingly.
- `GalleyPluginSpec.getDecorationRange()` still exists in `src/types.ts` and is used by `headings`, `blockquote`, `checkboxes`, and `dividers`.
- `code-fence.ts` and `tables.ts` use custom `StateField`s and do not depend on `makeBlockPlugin`.
- `GalleyRenderContext` currently carries `theme`, `mode`, and `codeHighlighter`; v0.6 adds `onLinkClick` and `imageRenderer`.
- `EditorController` owns dynamic extensions, so `bidi`, `onLinkClick`, and `imageRenderer` belong in `ControllerSettings` and are passed through `buildDynamicExtensions()`.

- [x] **Step 2: Add spec validation helper**

In `src/rendering.ts`, add:

```ts
function validatePluginSpec(spec: GalleyPluginSpec): void {
  const count = [
    spec.getLineRange,
    spec.getMarkRange,
    spec.getPointPosition,
  ].filter(Boolean).length;
  if (count > 1) {
    throw new Error('Only one of getLineRange, getMarkRange, or getPointPosition may be defined');
  }
}
```

Call it at the top of `makeInlinePlugin()` and `makeBlockPlugin()`.

- [x] **Step 3: Implement explicit range resolution**

Add helper functions:

```ts
function markRangeFor(spec, node, state) {
  return spec.getMarkRange?.(node, state) ?? { from: node.from, to: node.to };
}

function pointPositionFor(spec, node, state) {
  return spec.getPointPosition?.(node, state);
}

function lineRangeFor(spec, node, state) {
  return spec.getLineRange?.(node, state);
}
```

For inline factory:

- If result is `WidgetType`, require/use `getPointPosition()` when present, else `node.from`.
- If result is a `Decoration.line`, use `getLineRange()` and decorate every touched visible line.
- Otherwise use `getMarkRange()` or node range.

For block factory:

- If result is a widget, use `getPointPosition()` if present, else replace `[node.from, node.to]`.
- If result is a line decoration, use `getLineRange()` or node range and decorate every line in that range.
- Otherwise use `getMarkRange()` or node range.

- [x] **Step 4: Implement selective selection rebuilds**

In `makeInlinePlugin.update()` and `makeBlockPlugin.update()`:

```ts
const selectionChanged = update.selectionSet &&
  (spec.selectionAffectsDecorations?.(update.startState.selection, update.state.selection) ?? true);
```

For transactions in block fields, use `tr.startState.selection` and `tr.state.selection`.

- [x] **Step 3: Implement hook**

`useGalley(options?: { initialValue?: string; onChange?: (value: string) => void })` returns:

```ts
{
  ref: RefObject<GalleyHandle | null>;
  content: string;
  setContent(value: string): void;
  focus(): void;
  blur(): void;
  insertText(text: string): void;
  execCommand(name: BuiltinCommand | string, ...args: unknown[]): unknown;
  getContent(): string;
}
```

Implementation details:

- `const ref = useRef<GalleyHandle>(null)`.
- `content` is React state initialized from `initialValue`.
- `setContent` updates React state and calls `ref.current?.setContent(value)` when mounted.
- `onChange` is stored in a ref to keep command methods stable.
- All methods use `useCallback`.
