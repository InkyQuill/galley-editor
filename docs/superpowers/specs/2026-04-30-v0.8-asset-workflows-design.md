# v0.8 Asset Workflows Design

## Goal

Add consumer-owned file drop, paste upload, and image metadata workflows without turning Galley into a storage, upload, or asset-management layer.

## Scope

v0.8 covers two roadmap entries:

- File drop and paste upload hooks controlled by the consumer app.
- Image resize and asset metadata UX.

The milestone should ship as a focused editing workflow release after v0.7. It is still pre-1.0, so API names can change if the resulting contracts are clearer.

## Public API

Add a file intake hook that handles both paste and drop:

```ts
export type GalleyFileSource = 'paste' | 'drop';

export interface GalleyFileInput {
  files: File[];
  source: GalleyFileSource;
  event: ClipboardEvent | DragEvent;
  view: EditorView;
  selection: { from: number; to: number; anchor: number; head: number };
}

export type GalleyFileHandler =
  (input: GalleyFileInput) => string | string[] | null | false | Promise<string | string[] | null | false>;
```

`GalleyEditorProps` gets:

```ts
onFiles?: GalleyFileHandler;
```

When `onFiles` returns markdown, Galley inserts it at the active selection. `string[]` is joined with newlines. `null` or `false` means the consumer intentionally handled the event without insertion. If the handler throws or rejects, Galley leaves the document unchanged and calls a new optional error hook:

```ts
onFileError?: (error: unknown, input: GalleyFileInput) => void;
```

Existing `onPaste(event, view)` remains for raw clipboard inspection. `onFiles` runs only when clipboard/drop data contains one or more files. If both are present, `onFiles` gets the file workflow first; `onPaste` still receives the original paste event after file extraction so existing logging examples keep working.

## Drop Behavior

Galley registers `dragover` and `drop` handlers on the editor DOM. On `dragover`, it prevents default only when files are present and `onFiles` is configured. On `drop`, it computes the insertion position from CodeMirror's drop coordinates when possible, falls back to the current selection, calls `onFiles`, and inserts returned markdown at that range.

No upload progress UI is built in. Consumers can use toolbar/footer slots or app state to show progress.

## Paste Behavior

The existing paste event path is extended with file extraction from `event.clipboardData.files`. When files are present and `onFiles` is configured, the default browser paste is prevented and returned markdown is inserted after the promise settles.

Text-only paste stays unchanged.

## Image Metadata Model

Image parsing becomes a shared utility instead of being private to `src/plugins/images.ts`. The public renderer input expands to:

```ts
export interface GalleyImageInfo {
  alt: string;
  url: string;
  title?: string;
  width?: number;
  height?: number;
  raw: string;
  from: number;
  to: number;
}

export type ImageRenderer = (image: GalleyImageInfo) => HTMLElement | null;
```

Width and height are parsed from an optional attribute suffix after the image target:

```md
![Alt](image.png "Title"){width=640 height=360}
```

The suffix is Galley-authored markdown metadata, not a claim of full Markdown spec support. Unknown attributes are preserved in source when helpers update known fields.

## Image Editing Helpers

Add commands that update the current image markdown when the selection is inside or directly on an image token:

- `updateImageMetadata(input)` updates `alt`, `title`, `url`, `width`, and/or `height`.
- `clearImageDimensions()` removes width/height metadata.

The commands operate on markdown text only. They do not inspect image natural dimensions or perform file transformations.

Optional image resize UX is implemented by consumer code through `imageRenderer`. Galley provides the metadata parsing, command helpers, and Storybook examples showing resize handles that call `updateImageMetadata`.

## Storybook

Add examples for:

- paste upload: fake async upload returns `![name](uploaded/name)`;
- drop upload: dropped files insert image links at the drop position;
- image metadata editor: alt/title fields and width/height controls update markdown;
- custom image renderer with resize handles.

The examples must use in-memory fake uploads. No network, object storage, or server endpoint is required.

## Error Handling

File handler errors are non-destructive. Galley must not insert partial markdown after a failed promise. `onFileError` receives the error and original input. If `onFileError` is absent, errors are reported to `console.error` in development-friendly wording.

## Out Of Scope

- Bundled uploads, signed URL handling, storage adapters, or progress components.
- Dragging existing images to reorder them.
- Binary image transformation, compression, cropping, or EXIF parsing.
- Full Markdown attribute syntax beyond image width/height preservation.

## Acceptance Criteria

- Pasted file data can be intercepted and converted to markdown by a consumer callback.
- Dropped file data can be intercepted and inserted at the intended document position.
- Text-only paste behavior remains compatible with v0.7.
- `imageRenderer` receives alt, URL, title, source range, raw markdown, and optional dimensions.
- Commands can update image alt/title/URL/dimensions without rewriting unrelated document text.
- Storybook demonstrates fake paste upload, fake drop upload, metadata editing, and resize-handle rendering.
- Tests cover successful async insertion, null/false no-op, rejected handler no-op, text-only paste compatibility, drop position fallback, and metadata command edge cases.
