# v0.9 Advanced Authoring Design

## Goal

Make Galley Editor's table authoring feel like a polished visual editing experience while keeping Markdown as the source of truth.

## Scope

v0.9 covers one roadmap entry:

- Hybrid visual table editing for simple GFM pipe tables.

Tables are the release headline. The implementation must prioritize user experience over implementation simplicity: the rendered table widget should be editable without dropping into raw source, just like the v0.8 image workflow made images editable as visual objects.

KaTeX-style math and Mermaid-style diagram renderer examples are deferred to the next roadmap step. They should not compete with the table editing UX in v0.9.0.

## Product Behavior

In live mode, a parseable Markdown pipe table renders as a visual table widget. Clicking a cell selects it visually and keeps the table rendered. Typing starts editing that cell directly; the focused cell shows a compact raw-Markdown cell editor, while the rest of the table remains visual. This makes the default workflow feel familiar to users who know spreadsheet tools without turning Galley into a spreadsheet engine.

Keyboard behavior is Excel-like within a deliberately small scope:

- `Tab` commits the active cell and moves right.
- `Shift+Tab` commits the active cell and moves left.
- `Enter` commits the active cell and moves down.
- Arrow keys move between cells when the cell editor caret is at the corresponding edge.
- `Esc` cancels an active draft or defocuses a selected cell.
- Paste replaces the active cell content. Multi-cell paste and range fill are out of scope for v0.9.

Raw table Markdown remains available through explicit source escape: `Ctrl`/`Cmd` click, a Source control on the table widget, or keyboard navigation into the source range. Normal click should not reveal raw table Markdown. This mirrors the v0.8 image behavior: visual selection first, source only when the user asks for it or navigates there deliberately.

Inactive cells render basic inline Markdown visually, including emphasis, links, and inline code where practical. The active cell edits its raw cell Markdown so the user can adjust the source precisely.

## Architecture

The existing table-only parser in `src/plugins/tables.ts` should move into a shared table model module used by rendering, commands, and tests. The model must represent:

- the table source range;
- header, separator, and body rows;
- column alignments;
- normalized row and column counts;
- source spans for each cell;
- enough row/column identity to restore the selected cell after a table rewrite.

The model stays within the existing simple GFM pipe-table scope: one header row, one separator row, and zero or more body rows. It should preserve raw cell text where possible, normalize pipe spacing after structural edits, and return `null` for invalid or unsupported tables so Galley falls back to raw Markdown instead of rendering broken controls.

The table widget should become an interactive CodeMirror plugin surface. A table interaction `StateField` or `ViewPlugin` tracks the active table, active cell, edit draft, and source-mode escape state. DOM events in the widget dispatch effects or transactions for selecting a cell, starting edit, committing a draft, cancelling a draft, inserting or deleting rows/columns, changing alignment, and revealing source.

Cell commits rewrite the Markdown source through the shared serializer. They should restore the visual cell selection after the transaction and avoid unnecessary editor scrolling. The same command functions should power widget controls and consumer toolbar integrations so app-level buttons and the built-in widget cannot drift in behavior.

`preview` mode and `editable={false}` render tables visually with no editing state and no controls.

## Commands

Expose table commands that operate on the Markdown table containing the current visual or source selection:

- `normalizeTable()`
- `insertTableRowBefore()` and `insertTableRowAfter()`
- `deleteTableRow()`
- `insertTableColumnBefore()` and `insertTableColumnAfter()`
- `deleteTableColumn()`
- `setTableColumnAlignment(alignment: 'left' | 'center' | 'right' | null)`
- `revealTableSource()` if the widget needs a public command for Source controls and consumer integrations.

Commands return `false` when the selection is not inside a parseable table. Multi-cursor behavior applies each command once per distinct table. Structural commands should keep at least one header column and should not leave malformed separator rows.

## Widget Controls

The default table widget should include quiet, table-local controls that appear when the table is selected or focused:

- add row before and after;
- add column before and after;
- delete current row;
- delete current column;
- set current column alignment;
- reveal source.

Controls must be keyboard reachable and must have concise `aria-label` values. Their visual language should match the v0.8 image handles: present when the object is selected, restrained when idle, and absent in preview or readonly mode.

Consumer override hooks for the table controls renderer are useful, but they are not required for v0.9 unless they fall naturally out of the existing settings and renderer pattern. A polished default UX is more important than adding a customization API before the core behavior settles.

## Storybook

Add a “Visual Table Editing” story with realistic table content. It should demonstrate:

- selecting a rendered cell;
- editing a cell without revealing the whole Markdown table;
- keyboard navigation;
- row and column controls;
- alignment controls;
- source escape;
- disabled controls in preview and readonly modes.

## Documentation

Update docs to present 0.9.0 as the table UX release. Documentation should cover:

- the hybrid visual/source table editing model;
- Excel-lite keyboard behavior;
- command names, arguments, return values, and outside-table `false` behavior;
- Markdown table limitations;
- how consumer toolbar buttons can call the same table commands.

Remove or replace older wording that says direct cell editing is out of scope.

## Out Of Scope

- Bundling KaTeX, Mermaid, Prism, Shiki, or any renderer runtime.
- KaTeX-style math and Mermaid-style diagram renderer examples.
- Full Markdown parser extension support beyond existing CodeMirror/Galley plugin registration patterns.
- Multi-cell range selection.
- Rectangular paste across multiple cells.
- Fill handles.
- Row or column drag reordering.
- Spreadsheet formulas.
- Complex escaped-pipe table parsing beyond the existing GFM-style pipe table scope.
- A public table-controls renderer override unless it is low-risk and follows existing renderer settings cleanly.

## Acceptance Criteria

- A rendered table can be selected and edited visually in live mode without raw Markdown appearing on normal click.
- Cell edits commit back to Markdown source and preserve table structure.
- `Tab`, `Shift+Tab`, `Enter`, edge-aware arrow navigation, `Esc`, and single-cell paste behave predictably.
- Source escape works through `Ctrl`/`Cmd` click and a widget Source control.
- Inactive cells render basic inline Markdown visually, while active cells edit raw cell Markdown.
- Row, column, alignment, delete, and source controls are accessible, editable-only, and powered by the same command path as consumer integrations.
- Preview and readonly modes render tables visually with no editing affordances.
- Unsupported tables fall back to raw Markdown instead of rendering broken widgets.
- Table model and command tests cover parse/serialize, cell source spans, commit cell edit, keyboard movement, row and column operations, alignment changes, source fallback, outside-table `false` behavior, and duplicate-table protection for multi-cursor selections.
- Plugin tests cover selected cell, focused editor, commit/cancel, preview/read-only gating, and no scroll jump after commit.
- Docs and Storybook show the hybrid table workflow.
